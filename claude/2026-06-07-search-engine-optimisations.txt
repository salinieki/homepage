  SEO analysis — salinieki.co.uk

  Strong points

  - Solid technical foundation. Hugo emits semantic HTML (<main>, <article>,
  <nav>, single <h1> per page), lang attribute set per page, mobile viewport,
  lazy-loaded images, lazy-loaded iframes (Google Maps, Facebook). Pages are
  small and fast.
  - Real, original, keyword-rich content in Latvian — "latviešu tautas deju
  kolektīvs Londonā", named choreographers, dated concert history going back
  to 2012. This is exactly the kind of unique content Google likes; almost no
  other UK page covers this niche.
  - Bilingual setup (LV/EN) with per-language title and description in
  hugo.toml:14-50 and a visible language switcher
  (layouts/partials/language-switcher.html:19) emitting hreflang on the anchor.
  - Clean URLs and clean menu structure, sitemap + robots enabled (hugo.toml:5),
   RSS on home and sections.
  - Alt text on images, descriptive page titles, proper <title> template in
  baseof.html:18.
  - Internal linking via prev/next between news articles (single.html:24) helps
  crawl depth.

  Issues, ordered by impact

  1. The public/ checked into the repo was built by hugo serve, not hugo

  public/sitemap.xml points to http://localhost:1313/lv/sitemap.xml, and
  public/index.html includes the livereload.js script and <meta 
  name="generator"> only — no real meta. If this is what gets deployed, search
  engines see broken sitemap URLs and a dev build. Make sure the deploy pipeline
   runs plain hugo (or hugo --minify) against the production baseURL, and don't
  commit public/.

  2. No Open Graph / Twitter Card meta

  baseof.html:18-19 only emits <title> and description. Anything shared on
  Facebook, WhatsApp, Instagram DMs, or Slack will show no image and a bare URL.
   For a folk-dance group whose audience is on Facebook, this is the single
  biggest practical win. Add in <head>:

  <meta property="og:title" content="...">
  <meta property="og:description" content="...">
  <meta property="og:image" content="{{ absURL of cover/hero }}">
  <meta property="og:type" content="{{ "article" or "website" }}">
  <meta property="og:url" content="{{ .Permalink }}">
  <meta property="og:locale" content="lv_LV"> (or en_GB)
  <meta property="og:locale:alternate" content="en_GB">
  <meta name="twitter:card" content="summary_large_image">

  For news pages, prefer the article's coverImage; otherwise fall back to a hero
   shot or the logo.

  3. No hreflang in <head>

  Google's documented signal lives in <head>, not in body anchors. Add to
  baseof.html:

  {{ range .AllTranslations }}
    <link rel="alternate" hreflang="{{ .Lang }}" href="{{ .Permalink }}">
  {{ end }}
  <link rel="alternate" hreflang="x-default" href="{{ .Site.BaseURL }}">

  This is the missing piece that tells Google to surface the English page to UK
  searchers and the Latvian page to Latvian searchers.

  4. No canonical URL

  Add <link rel="canonical" href="{{ .Permalink }}"> in baseof.html. Cheap,
  prevents duplicate-content issues across /foto/, query strings, etc.

  5. No structured data (JSON-LD)

  Three low-effort wins for this site:

  - Organization / PerformingGroup on every page (name, logo, sameAs
  Facebook/Instagram/YouTube, foundingDate 2012).
  - Event for each upcoming entry in params.calendar (hugo.toml:19-21) —
  eligibility for Google's events rich results, which is unusually high-yield
  for a small site.
  - Place/PostalAddress for the rehearsal venue on the Kontakti page — helps
  "Latvian dance London" / local intent queries.

  6. English URLs use Latvian slugs

  /en/par-mums/, /en/foto/, /en/kontakti/ (hugo.toml:71-75). English speakers
  search "about us", "contact", "photos". Either translate slugs (/en/about/,
  /en/contact/, /en/photos/) or live with it — but the current setup loses the
  easy keyword match in URLs.

  7. robots.txt is essentially empty

  Current output is just User-agent: *. Add a Sitemap: line. Easiest: enable a
  custom layouts/robots.txt or rely on Hugo's default which does include it —
  verify what's actually shipped.

  8. Home page lacks keyword-rich <h1> and intro copy

  hero.html:17 is just <h1 class="hero-title">Salinieki</h1>. That's your one h1
   on the most important page. Consider <h1>Londonas latviešu tautas deju kopa 
  "Salinieki"</h1> (and the English equivalent on /en/). Also: the home page has
   no intro paragraph — just hero, news list, CTA. A 2–3 sentence intro under
  the hero would significantly help ranking for "Latvian folk dance London".

  9. No width/height on news card images

  news-card.html:13,17 — adding intrinsic dimensions prevents layout shift, a
  Core Web Vitals signal.

  10. Minor

  - Favicon is images/logo.png only — no apple-touch-icon, no favicon.ico, no
  manifest.json. Doesn't move SEO but affects how the site appears in browser
  tabs / home screens.
  - Footer has no NAP (name/address/phone) block — adding rehearsal address

  - Figure shortcodes use short generic alts ("Mēģinājums", "Gājiens"); slightly

  - categories/ and tags/ directories are generated but probably empty — empty

  - Google Fonts + jsdelivr CDN: fine for SEO, but self-hosting Nunito would

  1. Fix the build/deploy so public/ isn't a hugo serve artifact.
  2. Add Open Graph + canonical + head-level hreflang to baseof.html — one edit,
   big payoff.
  3. Add Organization + Event JSON-LD.
  4. Rewrite the home h1 and add an intro paragraph.

  Everything else is incremental.

⏺ What Open Graph is

  Open Graph (OG) is a tiny set of <meta> tags in <head> that tells social
  platforms and chat apps how to render a link preview. Twitter Cards are the
  equivalent for X/Twitter. Most other platforms (LinkedIn, Slack, Discord,
  iMessage, WhatsApp, Telegram, Signal, Facebook Messenger) read the OG tags.

  A minimal set looks like:

  <meta property="og:title"       content="10. Kultūras Diena Lesterā">
  <meta property="og:description" content="Maija sākumā Salinieki devās uz 
  Lesteri…">
  <meta property="og:image"       content="https://salinieki.co.uk/images/foto/2
  026-05-lestera-kulturas-diena/04-grupas-bilde.jpg">
  <meta property="og:url"         
  content="https://salinieki.co.uk/news/2026-05-05-lestere-kulturas-diena/">
  <meta property="og:type"        content="article">
  <meta property="og:site_name"   content="Salinieki">
  <meta name="twitter:card"       content="summary_large_image">

  What happens today when someone shares a link

  I just checked public/news/2026-05-05-lestere-kulturas-diena/ — there are no
  OG tags. So when someone pastes
  https://salinieki.co.uk/news/2026-05-05-lestere-kulturas-diena/ into Facebook,
   WhatsApp, Messenger, an Instagram bio link click, a Slack channel, etc., the
  platform's scraper falls back to:

  - Title: the <title> tag — usually fine
  - Description: the <meta name="description"> if present, otherwise the first
  plausible paragraph — often the wrong one (a sidebar line, a menu item)
  - Image: whatever the scraper picks. With no og:image, it usually grabs the
  first reasonably-sized image on the page. For your home page that might be a
  hero slide; for a news article it might be the lazy-loaded cover; for
  /par-mums/ it could be a leader headshot. Sometimes it picks the 64×64 logo.
  Sometimes it picks nothing and the preview is a grey rectangle.

  What you'd observe after fixing it

  Concrete, visible changes:

  1. Facebook posts and shares — when you (or anyone) shares a link to a news
  article on the Salinieki Facebook page, instead of a small thumbnail with a
  guessed title, you get a large card with the cover photo, the article title,
  and your chosen description. This is the format that gets clicks. For a folk
  dance group whose audience lives on Facebook, this is the single biggest
  payoff.
  2. WhatsApp and Messenger — when a member shares the link in a group chat, the
   preview shows the cover image and title instead of just the URL text. People
  are much more likely to tap a preview with a photo of dancers than a bare
  link.
  3. Instagram link-in-bio / Stories link stickers — the preview thumbnail comes
   from og:image.
  4. Slack / Discord / Teams — link unfurls show the cover and description
  rather than the URL.
  5. iMessage — same; you get a rich preview bubble with the cover photo.
  6. Search engines — Google does not use OG tags for ranking, but it sometimes
  uses og:title/og:description as fallbacks when its own snippet extraction is
  poor. Minor.

  How big is the impact really

  For a public-facing site whose distribution channel is "people sharing it on
  Facebook," moving from "no preview / wrong image" to "large branded cover
  image" is the kind of change that meaningfully lifts click-through. There are
  no hard numbers I can quote for your specific site, but in the industry, rich
  link previews typically lift CTR on social shares by a substantial multiple
  over bare-URL previews — particularly on mobile where the URL alone is just a
  tiny line of text.

  The cost is also tiny: ~10 lines in baseof.html, mapping og:image to
  .Params.coverImage for news pages and a default hero image for everything
  else. You can verify the result in the Facebook Sharing Debugger — paste a URL
   and see exactly what the preview will look like.

❯ Let's move on to point 2. - No canonical link. What does that mean?           

⏺ What a canonical link is

  A canonical link is a single line in <head>:

  <link rel="canonical" 
  href="https://salinieki.co.uk/news/2026-05-05-lestere-kulturas-diena/">

  It tells search engines: "this URL is the authoritative address for this 
  page." When several different URLs serve the same (or very similar) content,
  the canonical tag points to the one you want indexed and ranked. Everyone
  agrees the others are duplicates of it.

  Why duplicates happen even when you didn't plan any

  Even on a clean static site like yours, the same page is reachable through
  multiple URLs in practice:

  - https://salinieki.co.uk/par-mums/
  - https://salinieki.co.uk/par-mums (no trailing slash; some servers serve
  both)
  - https://www.salinieki.co.uk/par-mums/ (with www; depends on DNS/CDN setup)
  - http://salinieki.co.uk/par-mums/ (if HTTP isn't strictly redirected)
  - https://salinieki.co.uk/par-mums/?fbclid=AbCdEf123 — what Facebook appends
  when someone clicks a shared link
  - https://salinieki.co.uk/par-mums/?utm_source=newsletter&utm_campaign=spring
  - https://salinieki.co.uk/par-mums/?gclid=… — Google Ads click ID

  To a search engine, every distinct URL is potentially a separate page. Without
   a canonical, Google has to guess which one is "the real one" — and may split
  your ranking signals across all the variants, or pick a tracking-parameter URL
   as the canonical (which then shows up in search results looking ugly).

  With a canonical, every variant declares "I'm a copy of
  https://salinieki.co.uk/par-mums/" and Google consolidates everything onto
  that one URL.

  What about your bilingual setup?

  This is not a canonical case — /par-mums/ (LV) and /en/par-mums/ (EN) are
  different content in different languages. Those are handled by hreflang, not
  canonical. Each page's canonical points to itself; the hreflang alternates
  point to the translations.

  What you'd observe after fixing it

  Honestly: probably nothing visible day-to-day. This is a defensive measure,
  not a growth lever like Open Graph. The wins are quiet:

  - Facebook click-tracking URLs (?fbclid=…) don't leak into Google search 
  results. Right now, if a popular FB post drives traffic, Google may end up
  indexing …/news/foo/?fbclid=AbCdEf as the canonical version of that article.
  With a <link rel="canonical"> it consolidates to the clean URL.
  - Ranking signals consolidate. Backlinks pointing to slightly-different
  variants of the same page all contribute to the canonical URL's authority
  instead of being split.
  - No accidental duplicate-content penalty if at some point the site becomes
  reachable on both www. and bare domain, or both HTTP/HTTPS.
  - Cleaner Search Console reports — fewer "Duplicate without user-selected
  canonical" warnings.

  It's a one-line edit and there's basically no downside, so it's worth doing
  even though the upside is "insurance" rather than "growth".

  The change

  Just add this in layouts/_default/baseof.html <head>:

  <link rel="canonical" href="{{ .Permalink }}">

  .Permalink is Hugo's built-in for the absolute URL of the current page, so on
  /par-mums/ it renders https://salinieki.co.uk/par-mums/, on /en/par-mums/ it
  renders https://salinieki.co.uk/en/par-mums/, and so on.

  Want me to add it?

❯ Next, can you explain what you meant by "No structured data JSON-LD"?         

⏺ What JSON-LD structured data is

  It's a small block of JSON inside a <script type="application/ld+json"> tag
  that tells search engines, in machine-readable form, what kind of thing a page
   is about — not just the words on the page, but the entities: "this is an
  organization, here's its address, founding date, social profiles" or "this is
  an event, here's when, where, and what".

  The format is schema.org (a shared vocabulary developed by Google, Bing,
  Yahoo, Yandex) expressed in JSON-LD (the syntax Google recommends). Hugo
  doesn't generate any of this — you add a template snippet that emits the JSON
  from your existing front matter and site params.

  Example for your group, sitting in <head>:

  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "PerformingGroup",
    "name": "Salinieki",
    "alternateName": "Londonas TDK \"Salinieki\"",
    "url": "https://salinieki.co.uk/",
    "logo": "https://salinieki.co.uk/images/logo.png",
    "foundingDate": "2012",
    "location": {
      "@type": "Place",
      "name": "Daugavas Vanagu nams",
      "address": {
        "@type": "PostalAddress",
        "streetAddress": "72 Queensborough Terrace",
        "postalCode": "W2 3SH",
        "addressLocality": "London",
        "addressCountry": "GB"
      }
    },
    "email": "info@salinieki.co.uk",
    "sameAs": [
      "https://www.facebook.com/salinieki/",
      "https://www.instagram.com/salinieki/",
      "https://www.youtube.com/@saliniekitdk9717"
    ]
  }
  </script>

  Nothing visible to humans. Search engines read it.

  Why it matters

  Schema.org markup is what powers rich results in Google — the visually
  distinct search listings that aren't just "blue title + grey URL + snippet".
  The relevant ones for your site:

  1. Organization / PerformingGroup (every page, baseline)

  Builds a knowledge panel — the box that sometimes appears on the right side of
   Google results when someone searches "Salinieki" or "Salinieki London".
  Includes your logo, founding year, social links, location. Google often
  constructs these automatically but markup makes it much more reliable, gets
  the right logo, and ensures your social profiles (sameAs) are linked correctly
   so the panel shows Facebook/Instagram/YouTube icons.

  Visible change: a richer-looking entity in search results when people search
  your name, with the correct logo and social links instead of whatever Google
  scraped.

  2. Event (highest practical value for you)

  Your hugo.toml:19-21 already has a calendar entry:

  [[languages.lv.params.calendar]]
    date = "20.06.2026"
    title = "\"(Pa) Saules Ceļu Izdejot\" Īrijā"

  If that's emitted as Event JSON-LD on the relevant page, with startDate,
  location, performer, then Google can show your event as a rich event listing —
   a card with date, location, and a direct link. Events also flow into Google's
   dedicated "Events" search filter and into Google Maps' "what's happening
  nearby" surfaces.

  For a folk dance group whose upcoming performances are the most
  action-oriented thing on the site ("when can I go see them?"), this is the
  highest-ROI structured data you can add. The threshold for eligibility is low
  — you just need date, name, and location.

  Visible change: instead of "Salinieki — Latvian Folk Dance Group" as a generic
   blue link, search results for queries like "Latvian folk dance Ireland June
  2026" or "Salinieki concerts" could show a card-style result with date and
  venue. People searching for diaspora events can actually find yours.

  3. Article / NewsArticle for news posts

  Each /news/… page becomes eligible for richer treatment: prominent date,
  author, cover image in carousels, "top stories"-style surfacing for queries
  about specific events.

  Visible change: news entries are more likely to appear with the cover photo as
   a thumbnail in search results, with the publish date visible.

  4. Place / LocalBusiness for /kontakti/

  - No change to the human-visible site at all. It's pure machine readability.
